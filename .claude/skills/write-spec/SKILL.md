# スペック作成（公式 spec-driven）

Claude Code 公式の「インタビュー → SPEC.md を書く → 新セッションで実行」フローに従う（出典: code.claude.com/docs/en/best-practices）。中規模以上の機能・実験・構造変更の**着手前**に使う。

## 手順

1. **インタビュー** — 実装に必要な要件が揃うまでユーザーに質問する。曖昧なまま書き始めない。
2. **既存の正本を読む** — 関係する `docs/01〜08`・`docs/adr/`・対象コードを読み、重複させない箇所を把握する。推測でファイル名・関数名を書かず、実コードで確認する。
3. **SPEC.md を書く** — リポジトリ直下の `SPEC.md` に、今から着手する1機能の自己完結スペックを書く。公式の必須条件を満たす:
   - 関係する**ファイルとインターフェースを名指し**する。
   - **スコープ外**を明記する。
   - 末尾に**実行可能な E2E 検証ステップ**を置く。
4. **完結性チェック** — `SPEC.md` 単体で、文脈を持たない別セッションの Claude が実装できるか確認する。
5. **新セッションで実行** — スペックが完結したら、まっさらなセッションで `execute-task` により実装するようユーザーに促す。公式: *"Time spent making the spec precise pays off more than time spent watching the implementation."*

## ルール

- SPEC.md は「今着手する1機能」だけを持つ。完了したら確定内容を `docs/01〜08` / `docs/adr/` へ昇格し、SPEC.md は次の機能で上書きする。
- 正本の定義を `docs/01〜08` と重複させない。重複しそうなら本体へリンクする。
- 秘密情報・プライベートなパス・認証情報・個人の運用データを書かない。
- 小さく明確な修正（typo・log 追加・リネーム）はスペックを省いて直接行う（公式の例外）。
